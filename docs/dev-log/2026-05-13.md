# 2026-05-13 实施记录：按服务商适配思考程度

## 背景

用户在 Claude Code 中确认 Opus 4.6 可选择 `low`、`medium`、`high`、`max` 四档思考强度。项目原先在设置面板固定显示 `low`、`medium`、`high`、`xhigh`，并把该值直接作为 OpenAI 风格参数发送，这会导致 Claude / Gemini 的思考配置不准确。

## 实施内容

- 新增 `src/shared/reasoning.ts`，集中维护不同服务商和模型的思考档位、归一化规则和设置页说明。
- 设置面板改为根据当前 `apiProviderType` 和模型名动态显示可用档位；`off` 的界面文案改为“使用模型默认”，表示不额外发送思考参数。
- OpenAI-compatible 请求继续使用 `reasoning_effort` 或 Responses API 的 `reasoning.effort`，仅发送 `minimal`、`low`、`medium`、`high`、`xhigh`。
- Claude / Anthropic 请求改为：
  - Claude 4.6 / 4.7 / Mythos 使用 `thinking: { type: "adaptive", display: "omitted" }` 和 `output_config.effort`。
  - 旧 Claude 模型使用 `thinking: { type: "enabled", budget_tokens }` 回退映射。
  - Opus 4.6、Sonnet 4.6 和 Mythos 显示 `max`；Opus 4.7 显示 `xhigh` 和 `max`。
- Gemini 请求改为：
  - Gemini 3 使用 `generationConfig.thinkingConfig.thinkingLevel`。
  - Gemini 2.5 使用 `generationConfig.thinkingConfig.thinkingBudget`。
- 代理服务 `server/proxy-server.mjs` 同步实现同一套归一化和请求体生成逻辑，避免代理模式继续发送错误协议参数。
- 新增单元测试覆盖 OpenAI-compatible `minimal`、Claude Opus 4.6 `max` 和 Gemini 2.5 `thinkingBudget`。

## 后续注意

- 新增服务商协议或新模型思考档位时，优先更新 `src/shared/reasoning.ts`，再同步代理服务的归一化逻辑。
- 如果第三方聚合服务商对 Claude / Gemini 参数做了非官方转换，应优先通过 Makelove 这类 provider 测试直连和代理两条路径。
