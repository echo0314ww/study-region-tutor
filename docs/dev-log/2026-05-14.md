## P0：优先修复项

### 1. 统一环境变量优先级

**问题**

主进程读取环境配置时，外部 `process.env` 优先级高于 `.env` 文件；但 `server/proxy-server.mjs` 和 `server/ngrok-dev.mjs` 当前是后读取的 `.env` 文件覆盖外部环境变量。这样同一套配置在应用、代理服务和 ngrok 开发脚本中可能表现不一致。

**目标**

统一为：外部环境变量优先，`.env.local` 次之，`.env` 最后。这样命令行临时注入的配置不会被文件覆盖。

**涉及文件**

- `src/main/env.ts`
- `server/proxy-server.mjs`
- `server/ngrok-dev.mjs`
- 视实现情况新增 `server/runtime-env.mjs` 或等价共享配置模块
- 视实现情况新增 `tests/proxyRuntimeEnv.test.ts`

**实施步骤**

1. 抽出代理服务和 ngrok 开发脚本共用的环境文件加载函数。
2. 明确加载顺序：先读取 `.env`，再覆盖 `.env.local`，最后覆盖 `process.env`。
3. 保持主进程 `src/main/env.ts` 与代理脚本语义一致。
4. 给共享加载函数补单元测试，覆盖同名变量在三处来源同时存在时的最终值。
5. 检查文档中的配置说明，补充“命令行环境变量优先级最高”。

**验证**

- 设置 `.env`、`.env.local` 和命令行环境变量为不同值，确认最终使用命令行值。
- 运行 `npm run proxy:dev` 和 `npm run ngrok:dev`，确认两者读取结果一致。
- 执行 `npm run test`。

**风险与回滚**

已有用户如果依赖 `.env.local` 覆盖命令行变量，行为会变化。需要在 README 或配置文档中明确说明优先级；如出现兼容问题，可增加启动日志提示当前配置来源。

### 2. 代理服务完整热更新 `AI_PROVIDERS`

**问题**

当前已经修复了 `AI_PROVIDERS` 删除后代理仍沿用旧配置的问题，但文件监听仍主要依赖启动时已存在的文件。`.env.local` 被新建、删除、重命名或编辑器原子替换时，仍可能漏掉变化。

**目标**

代理服务启动后，`.env` 和 `.env.local` 的创建、删除、重命名、原子替换、内容修改都能触发配置重载。若配置无效，服务应立即进入不可用状态，并通过 `/health` 和 API 响应明确返回配置错误。

**涉及文件**

- `server/proxy-server.mjs`
- 视实现情况新增 `tests/proxyConfigReload.test.ts`
- `README.md`
- `docs/proxy-config.example.env`

**实施步骤**

1. 在监听具体文件的基础上，增加对项目根目录的目录级监听。
2. 只响应 `.env` 和 `.env.local` 的变化，避免无关文件导致频繁重载。
3. 保留 debounce，合并短时间内的多次 change/rename 事件。
4. 当配置文件消失或 `AI_PROVIDERS` 为空时，将 `activeConfig` 更新为不可用配置，而不是继续使用旧配置。
5. 启动日志、重载日志和 `/health` 返回值中明确显示当前配置状态。
6. 补充手动验证步骤：删除 `AI_PROVIDERS` 后 `/health` 应返回 503；恢复后应自动回到 200。

**验证**

- 启动 `npm run proxy:dev`。
- 删除 `AI_PROVIDERS` 行，确认模型请求失败且 `/health` 返回配置错误。
- 恢复 `AI_PROVIDERS` 行，确认无需重启即可恢复。
- 删除、新建、重命名 `.env.local`，确认均能触发重载。

**风险与回滚**

Windows 文件监听事件可能重复触发。需要 debounce 和状态对比，避免日志刷屏或重复重建配置。

### 3. 处理 `TUTOR_PROXY_PORT` 热更新与 ngrok 端口不一致

**问题**

代理服务运行中修改 `TUTOR_PROXY_PORT` 时，代理进程可能继续监听旧端口，而 `ngrok-dev` 重新读取后可能尝试转发到新端口，导致外网地址和本地代理端口不匹配。

**目标**

端口变更后行为必须明确。推荐支持代理服务自动重启监听端口；若实现成本或稳定性不合适，则至少在 `/health`、日志和 ngrok 脚本中明确提示“端口变更需要重启代理服务”。

**涉及文件**

- `server/proxy-server.mjs`
- `server/ngrok-dev.mjs`
- `README.md`

**实施步骤**

1. 在代理配置重载时检测旧端口和新端口是否不同。
2. 优先实现自动切换监听端口：关闭旧 server，使用新端口重新 listen。
3. 如果新端口监听失败，回退到旧端口并把配置状态标记为错误，避免静默不可用。
4. ngrok 脚本在检测到端口变化后重新检查本地 `/health`，确认目标端口可用再重启隧道。
5. 更新文档说明端口变更的实时行为。

**验证**

- 启动 `npm run proxy:dev` 和 `npm run ngrok:dev`。
- 修改 `TUTOR_PROXY_PORT`。
- 确认代理服务和 ngrok 最终指向同一端口。
- 确认旧端口不再被误认为可用。

**风险与回滚**

自动重启监听端口可能中断正在进行的流式请求。实现时要接受短暂中断，并在日志中明确提示。

### 4. 移除或弱化硬编码内置代理地址

**问题**

**目标**

生产构建不依赖固定 ngrok 地址。默认代理地址应来自运行时配置或用户设置；开发时如需默认代理地址，也应通过环境变量注入。

**涉及文件**

- `src/renderer/src/constants.ts`
- `src/renderer/src/App.tsx`
- `src/renderer/src/components/SettingsPanel.tsx`
- `README.md`

**实施步骤**

1. 将硬编码 `BUILT_IN_PROXY_URL` 改为空值，或改为读取构建期 `VITE_TUTOR_DEFAULT_PROXY_URL`。
2. 前端显示逻辑改为：有运行时默认值则显示默认值，没有则提示用户填写。
3. 设置页面保留手动代理地址输入。
4. 更新 README 的代理模式说明，避免把临时 ngrok 地址写成固定推荐值。

**验证**

- 无默认代理地址时，设置页不显示过期地址。
- 配置 `VITE_TUTOR_DEFAULT_PROXY_URL` 后，开发构建可读取该值。
- 代理诊断能准确提示当前地址来源。

**风险与回滚**

如果当前用户已经依赖内置地址，删除后需要手动配置代理地址。文档中应说明如何通过设置页或环境变量配置。

## P1：结构性改进项

### 8. 复用 API 协议处理逻辑

**问题**

主进程直连 API 和代理服务中都有 endpoint 拼接、模型列表探测、Responses/Chat Completions 分支、流式解析和错误转换等逻辑。重复实现会导致后续修复需要改两处，增加行为漂移风险。

**目标**

建立共享的 API 协议层，至少让端点生成、模型列表候选路径、请求体构造、错误归一化和流式事件解析有统一测试。

**涉及文件**

- `src/main/openaiClient.ts`
- `server/proxy-server.mjs`
- 视实现情况新增 `src/shared/apiProtocol.ts`
- 视实现情况新增 `tests/apiProtocol.test.ts`

**实施步骤**

1. 先添加协议行为快照或表驱动测试，锁住现有兼容行为。
2. 抽取纯函数：端点拼接、模型列表候选路径、请求体构造、错误摘要。
3. 保留网络请求部分在各自运行环境内，避免一次性大迁移。
4. 第二阶段再评估是否统一流式 SSE 解析。
5. 每抽一组函数就跑直连和代理两条路径的测试。

**验证**

- 直连模式和代理模式对同一 provider 的 endpoint 完全一致。
- 模型列表失败时，两条路径给出的诊断建议一致。
- 流式输出不丢 chunk、不重复结束事件。

**风险与回滚**

这是中等风险改动。应分批抽取，不要和 provider 行为调整混在同一次修改里。

### 9. OCR 性能和稳定性优化

**问题**

OCR 初始化和识别可能较慢。频繁截图时，如果每次都重新初始化或没有明确的 worker 生命周期，会影响响应速度和内存占用。

**目标**

优化 OCR worker 生命周期，减少重复初始化成本，同时避免长期占用过多资源。

**涉及文件**

- `src/main/ocr.ts`
- `src/main/index.ts`
- 视实现情况新增 OCR 相关测试或手动验证记录

**实施步骤**

1. 梳理当前 OCR worker 创建、复用和释放逻辑。
2. 按语言缓存 worker，设置空闲释放时间。
3. 识别前检查语言数据和 worker 状态，失败时给出可诊断错误。
4. 避免多个 OCR 请求并发复用同一个不可并发对象；必要时排队。
5. 增加日志区分初始化耗时和识别耗时。

**验证**

- 第一次 OCR 和第二次 OCR 的耗时有明显区分。
- 连续多次截图不会出现 worker 状态错乱。
- OCR 关闭时不初始化 worker。

**风险与回滚**

worker 缓存可能增加常驻内存。需要空闲回收机制，并保留禁用 OCR 的设置路径。

## P2：安全和发布质量改进项

### 10. Renderer 安全加固

**问题**

当前 Electron renderer 需要继续检查 `sandbox`、CSP、preload 暴露面和外链打开策略。即使当前应用是本地工具，也应减少 renderer 被注入后触达主进程能力的风险。

**目标**

在不破坏现有截图、设置和 IPC 功能的前提下，逐步收紧 renderer 安全边界。

**涉及文件**

- `src/main/index.ts`
- `src/preload/index.ts`
- `src/renderer/index.html`
- `src/shared/ipc.ts`

**实施步骤**

1. 审计 preload 暴露的 API，确认只暴露必要方法。
2. 为 renderer 增加合适的 CSP。
3. 评估 `sandbox: true` 的可行性；如暂不能启用，在文档中记录阻塞点。
4. 检查外部链接统一通过 shell openExternal，避免 renderer 直接导航。
5. 增加安全配置回归清单。

**验证**

- 应用主要功能不受 CSP 影响。
- preload API 类型检查通过。
- 外链仍能正常打开，但不会在 Electron 窗口内导航。

**风险与回滚**

CSP 或 sandbox 可能影响开发热更新和部分依赖脚本。需要先在开发构建和生产构建分别验证。
